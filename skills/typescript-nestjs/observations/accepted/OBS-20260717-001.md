# No rule/checker coverage for raw numeric HTTP-status literals outside decorator strings

`references/http-boundary.md:13-15` says "status codes come from constant
registries, not string literals scattered through decorators", and `:49`
uses the type `Map<ErrorName, HttpStatus>` in an example — but the rule text
as written only names **string** literals in **decorators**
(`@HttpCode('...')`-style), and the example's `Map<ErrorName, HttpStatus>`
does not make explicit that the *value* type `HttpStatus` (from
`@nestjs/common`) is the canonical numeric-status registry that numeric-keyed
maps, thrown exceptions, and status assertions should also route through.
`scripts/check_nest_conventions.py` has no rule for raw numeric HTTP-status
literals in any of these three positions — confirmed by reading the checker
source: only `NEST-DI-TOKEN`, `NEST-RAW-THROW`, `NEST-DOMAIN-IMPORT`,
`NEST-APP-IMPORT` exist.

## Trigger observed in a consumer project

`apps/backend-api/common/http/problem.ts` built `STATUS_TO_PROBLEM_CODE` as
`new Map<number, ProblemCode>([[400, ...], [401, ...], ..., [503, ...]])` —
nine raw three-digit numeric literals as map keys — while the file already
imported `HttpStatus` from `@nestjs/common` and left it unused
(`@typescript-eslint/no-unused-vars` fired at the same time). Multiple
`__test__/**` spec files independently asserted `expect(...).toBe(404)` /
`.toBe(200)` instead of `HttpStatus.NOT_FOUND` / `HttpStatus.OK`.

## Reproduction (project-independent)

In any NestJS project, write `throw new HttpException(body, 404)`,
`@HttpCode(204)`, or `const STATUS_MAP = new Map<number, X>([[404, X.notFound]])`
instead of routing through `HttpStatus.*`; `check_nest_conventions.py` does
not flag any of the three, and `SKILL.md`/`references/http-boundary.md` give
no explicit example covering the numeric-literal case (only the
decorator-string case is named). See `data/fixtures/raw_http_status_literal.ts`
for a minimal, deterministic, project-independent fixture demonstrating all
three positions plus the equivalent clean forms.

## Suggested fix (delta-sized, pending review)

MINOR bump: a new checker rule `NEST-HTTP-STATUS-LITERAL` flagging a raw
numeric literal drawn from the standard HTTP status-code registry when it
appears as (a) a `@HttpCode(...)` decorator argument, (b) the status
argument of `new HttpException(...)`, (c) the key of a `[<code>, <value>]`
status-map/array entry, or (d) inside `.toBe(...)`/`.toEqual(...)` in a test
file (`*.spec.ts` etc.) — each guarded by negative tests so identifier-based
forms (`HttpStatus.NOT_FOUND`), non-status three-digit numbers, and
number/number pairs stay clean. Mirror the outcome as an explicit example in
`references/http-boundary.md`.
