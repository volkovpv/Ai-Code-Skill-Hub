# TS-SUPPRESS leaves no green path for a justified single-rule line disable

`scripts/check_conventions.py` reports `TS-SUPPRESS` for every
`eslint-disable` token, and the rule is deliberately excluded from the
checker's only false-positive mechanism (`skill-check-ignore` aborts with
exit 2 for `TS-SUPPRESS`). The prose rule mirrors this:
`references/typing-and-style.md` — "Do not silence the type checker or linter
(`@ts-ignore`, `@ts-nocheck`, `eslint-disable`) to pass a gate; fix the
cause."

That closes every path for a case where the narrow disable IS the correct
engineering outcome: a documented upstream limitation of a lint rule. Concrete
trigger observed in a consumer project (news-intel control-plane, strict
typescript-eslint flat config): `@typescript-eslint/promise-function-async`
with `{ allowAny: false }` reports every function whose declared return type
is `any`/`unknown`, because it cannot prove the value is not a Promise — a
limitation documented by typescript-eslint itself. Generic `unknown ->
unknown` helpers (a zod preprocessor, a recursive log-redaction walker) are
genuinely synchronous, so the code carries narrow, single-rule, justified
`// eslint-disable-next-line @typescript-eslint/promise-function-async -- <reason>`
lines. All authoritative gates (lint, typecheck, tests) are green on that
code, yet the skill self-check can never be: exit 1 with the disable, exit 2
with a `skill-check-ignore: TS-SUPPRESS` attempt, and every "fix" implied by
the rule text is strictly worse (disable the lint rule project-wide, weaken it
with `allowAny: true`, lie in the signature, or mark a synchronous function
`async`).

Occurrences: 4 findings across 2 files from 2 independent consumer tasks
(RM-TASK-005: `libs/config/env.validator.ts:39`; RM-TASK-006:
`libs/observability/log-redaction.ts:53,60,82`).

Reproduction (project-independent, Hub-side only — the fixture is
development-only content and is not shipped in a `runtime` install, see
`data/README.md`): run the checker over
`data/fixtures/justified_rule_disable.ts` — one `TS-SUPPRESS` finding, exit 1,
although the file holds a single-rule, line-scoped, justified disable working
around the documented rule limitation. Adding
`// skill-check-ignore: TS-SUPPRESS -- ...` to the same file aborts the check
with exit 2 ("TS-SUPPRESS can never be suppressed"). A consumer of a
`runtime` install can reproduce the same finding via the skill's own test
suite instead: `__test__/skills/test_typescript_coding.py::TestFixtureContract::test_justified_rule_disable_fixture_is_clean`.

Suggested consequence (pending review, delta-sized): narrow the `TS-SUPPRESS`
pattern so it does not fire on a line-scoped `eslint-disable-next-line` /
`eslint-disable-line` naming exactly one rule and carrying a non-empty `--`
justification; keep firing on `@ts-ignore`, `@ts-nocheck`, `eslint-disable`
without a rule code, multi-rule disables and unjustified disables — each
guarded by a negative test. Mirror the outcome in
`references/typing-and-style.md`.
