## Context (input)

Routine use of the installed `python-coding` skill: an agent following the skill's own calibration
pointers — «see `../data/fixtures/clean_sample.py` for a clean sample», «calibration in
`../data/fixtures/violations.py`» — to decide how strict a rule is meant to be.

The skill is installed with `install_mode: runtime` (`.agent-skills.lock.yaml`). The shipped tree is:

```
.agents/skills/python-coding/
├── data/
│   ├── README.md          ← names `fixtures/` as an existing sibling of `examples/`
│   └── examples/          ← the ONLY data directory actually shipped
│       ├── checked_input.py
│       └── checked_input.expected
├── knowledge/ references/ scripts/ agents/ observations/
```

## Expected behavior

A pointer inside shipped skill content resolves in the tree the consumer actually receives. Either
the target ships, or the shipped prose does not link to it. `data/README.md` states the contract as
a fact of the installed skill: «`fixtures/` feeds the automated tests of this skill; `examples/`
shows a minimal input/expected-output pair» — a reader has no way to learn that only one of the two
halves is present in a runtime install.

## Actual behavior

First upstream failure, as data: **7 references to `../data/fixtures/*` across 3 shipped files
resolve to nothing** (14 raw occurrences counting the duplicated link text). No `data/fixtures/`
directory exists in the installed tree.

| Shipped file | Line(s) | Target |
|---|---|---|
| `knowledge/patterns.md` | 28, 37, 58, 86 | `../data/fixtures/clean_sample.py` |
| `knowledge/patterns.md` | 106 | `../data/fixtures/violations.py` |
| `knowledge/pitfalls.md` | 31 | `../data/fixtures/masked_literals.py` |
| `references/typing-and-style.md` | 129 | `../data/fixtures/clean_sample.py` |

The pattern is not a typo but a **packaging boundary**: `fixtures/` is development-only content that
the runtime install strips, while the prose that cites it ships. `data/README.md` describes the
pre-strip tree, so it is wrong for every consumer of a runtime install.

## Minimal reproduction

Self-contained, project-independent — no repository content involved:

```sh
uv run skillctl install python-coding --target <any-empty-dir> --install-mode runtime
cd <any-empty-dir>/.agents/skills/python-coding
grep -ro 'data/fixtures/[A-Za-z_]*\.py' . | sort -u   # → 3 distinct targets
ls data/                                              # → README.md examples   (no fixtures/)
```

Every listed target is absent. Expected: either the grep returns nothing, or every target resolves.

## Universality check

Independent of any local norm, language version, framework or project layout: the defect is entirely
inside the skill's own shipped tree and reproduces on an empty target directory. It touches no
project threshold, no project boundary and no local rule, so SFL-INV-15 does not apply — nothing here
is a `harness/rules/*` concern. The identical defect ships in `typescript-coding`
([[OBS-20260721-002]]), which makes it a packaging-convention gap of the Hub rather than a slip in one
skill.

SFL-INV-08 threshold: met twice over — 7 independent occurrences across 3 files, **and** a
deterministic, project-independent minimal reproduction.

## Suggested fix (optional, delta-sized, non-binding)

Pick one, Hub-side, tests-first (a local edit is forbidden — SFL-INV-01):

1. **Ship the fixtures** in the runtime install (they are small, synthetic and secret-scanned per
   `data/README.md`) — the pointers then resolve as written; or
2. **Re-point the prose at `data/examples/`**, which does ship, and correct `data/README.md` so it
   describes the runtime tree rather than the development tree; or
3. **Strip the pointers** on package and keep the calibration text self-contained.

Whichever is chosen, the regression that must go red first is a **link-resolution test over the
packaged tree** — it is the absence of that test, not the individual links, that lets the class
recur on every skill the Hub publishes.
